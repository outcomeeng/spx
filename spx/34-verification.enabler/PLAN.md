## Program: test verification recording and claim

This is the authoritative, all-slice plan for making `test` a first-class verification type whose run is recorded in the run journal ‚Äî streamed live through a custom Vitest reporter ‚Äî and folded into the node status claim. The claim-side detail is cross-referenced from `spx/31-spec-domain.enabler/21-node-status.enabler/PLAN.md`, which points back here.

### The corrected model (why this program exists)

- **Verification is the five types** ‚Äî validate, test, evaluate, audit, review. `test` is a first-class verification type, not a separate track. The current split (deterministic `spx test`/`validation` walled off from the verification-run recording) is a relic; `spx/34-verification.enabler/verification.md` line 21 and `spx/60-surfaces.enabler/21-cli-surface.enabler/13-verify-command-surface.pdr.md` still encode that split and are slated to change.
- **The run journal is the universal evidence substrate; the snapshot store is a derived projection.** Every verification run ‚Äî deterministic (`validate`, `test`, `eval`) and agentic (`audit`, `review`) ‚Äî records **scope + finding + terminal** events into the append-only run journal (`spx/18-state.enabler/71-appendable-journal-store.enabler`), streamable to a backend before the run completes. **spx drives** the deterministic runs; an **agent drives** the agentic ones; both append through the same evidence ports. A run produces scope (the units it covers) and findings (the problems it surfaces) *regardless of verdict mode* ‚Äî a validation **stage**+**problems**, a test **module**+**failing cases**, an eval **case**, an audit/review **finding**. Whether the adapter emits those events as they occur or batched at the end is an **adapter property, not a type property**: `src/validation/registry.ts` streams its ordered stages natively, and a **custom Vitest reporter** (`onTestModuleStart/End`, `onTestCaseResult`, `onTestRunEnd`) streams test just as natively. The **snapshot store** (`spx/18-state.enabler/71-snapshot-store.enabler`) is a **derived** whole-payload projection folded *from* the journal for fast `spec status`/last-run reads (`spx.product.md`: "derived snapshots") ‚Äî **not** any type's primary evidence, and off the test evidence path. Its `PROVIDES` line still names "a test verification run's per-case snapshot, without appending an event history," which encodes the discarded snapshot-shaped-test framing and is reworded to the derived-projection role (tracked below).
- **Claim vs. verification.** The committed `spx.status.json` is a **claim** authored by whoever pushes; **CI is the arbiter** that reproduces claimed-passing evidence and refutes regressions. `spx spec status --update` **folds** available local evidence into the claim and **executes nothing**. The present behaviour ‚Äî `--update` re-running per-node tests when recorded evidence is stale/failing/absent (`spx/31-spec-domain.enabler/54-spec-cli-commands.enabler/21-status-testing-delegation.adr.md`) ‚Äî is the relic that produces the whole-tree serial re-run (observed: one failing test ‚Üí 60 serial vitest processes, ~7m42s vs a 2m07s batch). It is replaced by fold-only.
- **The spec‚Üítest‚Üísource graph is the addressing spine** (`spx/25-outcomeeng.enabler/31-spec-tree.enabler/21-graph.enabler`: `21-spec`, `32-test`, `43-source`). The test graph maps which test file is evidence for which node (claim addressing); the source graph derives staleness from spec-linked tests plus provider facts, **not language import graphs alone** (`.../11-graph-library-boundary.adr.md`, `21-graph.enabler/graph.md`). Today discovery/changed-set uses the TypeScript import-closure walker (`src/test/languages/typescript.ts`) ‚Äî the "garbage" the graph replaces. This program **reuses that discovery as-is and adds nothing to it**; the graph is a later slice.

### Slice 1 ‚Äî journal-native test vertical (TypeScript)

**Demonstrable value:** the spx-driven `test` type command-path (a noun ‚Äî grammar per *Execution surface* below) drives vitest programmatically with a **custom reporter** that streams **scope** events (per test module) and **finding** events (per failing case) into the run journal *as they resolve*, sealing with a **terminal** event ‚Äî the run is observable mid-flight through `spx verification run status/render`. `spx spec status --update` **folds** real per-reference outcomes from the journal into `spx.status.json` **executing nothing** ‚Äî a passing and a failing test file in one node resolve independently.

Custom-reporter evidence: Vitest's `Reporter` interface (`packages/vitest/src/node/types/reporter.ts`) exposes `onTestRunStart(specifications)`, `onTestModuleStart/End(testModule)`, `onTestCaseResult(testCase)`, and `onTestRunEnd(testModules, errors, reason)`. These map directly onto the journal event model ‚Äî module = scope unit, failing case = finding, `reason` = terminal status ‚Äî and fire **live as cases resolve**, so test streams natively with no `--reporter=json` capture and no conversion. The built-in `json` reporter's batch-at-finish behavior was never the type's nature; it was an adapter limitation this custom reporter removes.

Node set (`/apply` order; existing = reuse, NEW = compose):

1. **NEW** custom Vitest reporter ‚Äî an injected `Reporter` translating `onTestModuleStart/End` / `onTestCaseResult` / `onTestRunEnd` into journal scope/finding/terminal appends through the recorder's evidence ports. Placement (`spx/41-test.enabler` vs. the `test` type node) settled in `/decompose`.
2. **NEW** `test` **executor** ‚Äî the spx-driven type-noun handler in the new `src/commands/verification-exec/` that starts a verification run, drives vitest programmatically with the custom reporter injected, and finishes/seals the run. Registers the `test` type's **scope + finding + terminal** validators in the verify recorder's evidence-validator registry (`spx/34-verification.enabler/32-verify.enabler/13-verify-module-structure.adr.md`), exactly like every journal-shaped type.
3. **REUSE/extend** `spx/41-test.enabler/21-typescript-test.enabler` ‚Äî the vitest adapter runs programmatically with an injected reporter (not `--reporter=json --outputFile`), exposing the run to the reporter's hooks.
4. **NEW/extend** `spx/60-surfaces.enabler/21-cli-surface.enabler` ‚Äî add the spx-driven `test` type command-path (noun grammar per the table below) beside `spx verification run`; amend `13-verify-command-surface.pdr.md`.
5. **REUSE/extend** `spx/31-spec-domain.enabler/21-node-status.enabler` ‚Äî additive fold-from-journal path in `--update` (beside the existing path); maps test file ‚Üí node via existing spec-tree evidence references.

A derived-projection store still addresses `{scope}/{domain}/runs/` and takes a domain distinct from `test` (never `STATE_STORE_DOMAIN.TEST`, which `src/test/run-state.ts` occupies). This is its own spec change routed to `main`, not a slice-1 test-path node.

### Execution surface ‚Äî record vs. execute, and the noun grammar

Every verification kind lives under `spx verification`, so `spx test` and `spx validation` retire into it. Two things are distinct:

- **Who drives.** `spx verification run <verb>` is the **caller-driven** lifecycle: an external agent (for `audit`/`review`) drives the run and appends evidence through `start` / `input` / `scope add` / `finding add` / `finish` / `status` / `render`. The spx-driven deterministic types (`validate` / `test` / `eval`) have spx drive the run instead. Both record into the **same run journal**.
- **The command grammar.** `run` is a **noun** (the verification-run resource) carrying lifecycle **verbs**. Per `spx/60-surfaces.enabler/21-cli-surface.enabler/15-command-surface-governance.enabler` (noun groups for managed resources, user-intent verbs for caller actions), the spx-driven type command-paths must be **nouns too, not verbs** ‚Äî `spx verification validate` / `eval` as verb-siblings of the `run` noun is malformed. The exact noun scheme (e.g. type-nouns `validation` / `test` / `evaluation` beside `run`, versus another arrangement) is the open **`/decompose`+PDR** question; the fixed constraints are: nouns not verbs, and both the caller-driven and spx-driven paths record into the same run journal.

**Layering ‚Äî the executor is a client of the recorder.**

- `src/interfaces/cli/verify.ts` ‚Äî the `spx verification run` descriptor, **unchanged**; the only module parsing Commander for the run family, per `13-verify-module-structure.adr.md`.
- the spx-driven type command-paths parse through the executor's own CLI boundary ‚Äî **not** folded into `verify.ts`. How the shared `spx verification` parent composes the run descriptor and the type command-paths is the `/decompose`+PDR noun-grammar question above.
- `src/commands/verify/` ‚Äî the recorder, pure run lifecycle over the journal, **unchanged**.
- `src/commands/verification-exec/` ‚Äî the executor: drives `src/test/registry.ts` (with the custom Vitest reporter) / `src/validation/registry.ts` and appends scope/finding/terminal events through the injected recorder ports.

Both `13-verify-module-structure.adr.md` invariants stay literally true, so it needs no amendment: "verifier execution remains outside the verify module" (execution lives in the exec module; the verify module still only records) and "`src/interfaces/cli/verify.ts` is the only verify module that parses Commander" for `spx verification run` (the run descriptor is untouched; the executor parses through its own boundary).

**Every type journals scope + finding; streaming granularity is an adapter property.** A verification run appends one event per progress unit ‚Äî a validation **stage**+**problems** (`src/validation/registry.ts` exposes ordered pipeline stages, the `[1/7]‚Ä¶[7/7]` output), a test **module**+**failing cases** (via the custom reporter), an eval **case**, an audit/review **finding**; `verification.md` asserts a run "streams to its backend incrementally as events append." All five register **scope + finding + terminal** validators. Streaming *granularity* is the adapter property: validation streams its stages natively, the custom Vitest reporter streams its cases natively, and any adapter that can only emit a terminal batch appends its units at `finish` ‚Äî degraded observability under the same journal contract, never a different substrate.

**Retirement is an equivalence gate, not a judgment call.** Nothing retires until a per-reference **equivalence test** shows the executor's outcomes match the retiring surface's over a fixture corpus ‚Äî the `test` type command-path vs `spx test`'s `TestRunState`, the `validate` type command-path vs `spx validation`. Until then the surfaces coexist. "Fully baked" becomes a passing assertion.

### Slice 2 ‚Äî authored lifecycle status; retire `spx/EXCLUDE`

**Demonstrable value:** `spx.status.json` carries an **authored** lifecycle status (`declared`/`specified`) ‚Äî a hand-written "intentionally not passing yet" marker distinct from the machine-folded per-reference evidence outcomes. A node bearing it is **not discovered or run**, **not recorded as failing**, and **honored by CI** (passes the gate without its tests running). This lets authors write spec + test files without an implementation, keeps CI green, wastes no time discovering tests whose implementation does not exist, and **removes `spx/EXCLUDE`** ‚Äî the per-node claim scopes the gate in its own committed record.

Detail (owned by `spx/31-spec-domain.enabler/21-node-status.enabler/PLAN.md`, summarized here):

- **Status-file contract** (`spx/31-spec-domain.enabler/21-node-status.enabler/15-status-file-contract.pdr.md`): add an authored lifecycle field beside the machine-folded evidence outcomes. Two concerns ‚Äî the author owns the lifecycle intent; `--update` owns the evidence.
- **Fold:** `--update` folds per-reference evidence but **never overwrites the authored lifecycle status**; a `specified` node stays `specified` when evidence is absent or failing, never downgraded to `failing`.
- **Verification run / discovery:** skip nodes authored `declared`/`specified` (no wasted discovery, no failing record for absent implementations).
- **CI:** reproduces/refutes only nodes whose authored status expects passing; `declared`/`specified` nodes are not run and pass the gate.
- **Remove `spx/EXCLUDE`**; update the `CLAUDE.md` no-hand-edit rule so the lifecycle field is authorable while `passed`/`failed`/`not-run` evidence values stay machine-only.
- Governed by `15-status-file-contract.pdr.md` + `21-node-status-architecture.adr.md`. Reconcile with node-status's own evacuation plan (that node is slated to shrink; the authored-status contract may land in its future backend/spec-tree home).
