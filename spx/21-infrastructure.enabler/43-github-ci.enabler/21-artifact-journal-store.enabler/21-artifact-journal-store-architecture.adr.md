# Artifact Journal Store Architecture

GitHub artifact transport for agentic run journals is owned by the verification workflow's `actions/upload-artifact` and `actions/download-artifact` steps; the process performs no programmatic artifact I/O. During a job the GitHub Appendable journal backend is the runner-local JSONL appendable store of `spx/18-state.enabler/71-appendable-journal-store.enabler` unchanged — `append`, `readAll`, `seal`, and `isSealed` operate the local run file with no network or Actions-runtime access — and a sealed run's durable cross-job record is that file, uploaded by the workflow under the per-run artifact name `spx-journal-pr-<pull>.<type>.<run-token>` addressed by pull request, verification type, and run token. Opening a run hydrates the pull request's prior runs by reading the run files the workflow's download step restored into the runs directory, identifying them by that same pull-request-and-type naming.

## Rationale

A GitHub Actions artifact cannot be uploaded into a running job through the REST API or `gh`; only the Actions toolkit reaches the artifact service, a dependency the product does not carry. The product's artifact mechanism is the declarative, SHA-pinned `actions/upload-artifact` / `actions/download-artifact` step — `.github/workflows/deterministic-verification.yml` persists the `dist` build across jobs exactly this way — so durable artifact movement is a workflow concern, and a run journal is one more directory the workflow uploads at the end of a job and restores at the start of the next.

Locating transport in the workflow holds the product's offline-first core (`spx/spx.product.md`: never require network access for core operations): the workflow, not `spx journal`, touches the Actions artifact service, so the command stays a local-file program that runs offline and outside Actions. Reusing the runner-local appendable store keeps one JSONL format and one append/readAll/seal/replay contract across the local and GitHub backends; the only GitHub-specific concern is the run-scoped artifact name, which the workflow's steps and the hydration read share as their sole contract. Addressing each run's artifact by pull request and run token makes the pull request's source of truth the union of its restored run files, which concurrent jobs extend without collision.

Rejected: a process-level Actions-artifact client that uploads, lists, and downloads from inside the Node process — it forces the heavyweight Actions toolkit dependency the product avoids, breaks the offline-first core by making the command reach the artifact service, and duplicates the transport the workflow already performs declaratively. Rejected: making the artifact the per-append target — an artifact is immutable once written and unavailable mid-write, so it cannot back `append`; the runner-local file is the append target and the uploaded artifact its sealed snapshot. Rejected: persisting a run only as its pull-request-comment projection — a projection is a rendered view, not the event history, so it can neither replay nor resume a run; the Snapshot sink of `spx/21-infrastructure.enabler/43-github-ci.enabler/21-snapshot-adapter.enabler` owns the comment, this contract owns the durable events.

## Invariants

- The journal backend the process binds is the runner-local JSONL appendable store of `spx/18-state.enabler/71-appendable-journal-store.enabler` unchanged; `append`/`readAll`/`seal`/`isSealed` preserve the journal contract and touch only the runner filesystem.
- A restored run file holds exactly that run's sealed JSONL event history; re-reading it replays the identical events in ascending `seq` order.
- The contract between the workflow's artifact steps and the process's hydration read is the per-run artifact name `spx-journal-pr-<pull>.<type>.<run-token>` together with the run file residing inside the restored artifact directory at the run file's own basename; a pull request's readable run set is the union of the restored artifacts whose name carries its pull-and-type prefix.
- Constructing the backend reads no ambient Actions environment and performs no network access.

## Verification

### Audit

- ALWAYS: the GitHub Appendable journal backend binds the runner-local JSONL appendable store of `spx/18-state.enabler/71-appendable-journal-store.enabler` without widening or altering the `append`/`readAll`/`seal`/`isSealed` contract, per `spx/15-agent-run-journal.enabler/32-journal-module-structure.adr.md` ([audit])
- ALWAYS: persisting a sealed run and restoring a pull request's prior runs across the ephemeral runner is performed by the verification workflow's `actions/upload-artifact` and `actions/download-artifact` steps, mirroring the `dist` persistence of `.github/workflows/deterministic-verification.yml` ([audit])
- ALWAYS: the per-run artifact name addressed by pull request, verification type, and run token is the shared contract between the workflow's artifact steps and the process's hydration read, so concurrent jobs retain disjoint artifacts and hydration materializes only the pull request's own verification type ([audit])
- ALWAYS: opening a run identifies the pull request's prior runs by reading the run files restored into the runs directory, routing the runner filesystem through an injected `StateStoreFileSystem` per `spx/17-state.adr.md`, so the hydration read verifies over a controlled filesystem at `l1` without a network or the Actions runtime ([audit])
- NEVER: this backend performs programmatic GitHub artifact I/O for a run's durable event history — it imports no Actions-artifact toolkit, constructs no `gh` artifact API request, and binds no injected Actions-artifact upload/list/download client; the backend's I/O is runner-local-file, and the event-history artifact movement is the workflow's ([audit])
- NEVER: an artifact backs `append` as a per-event store, or a rendered projection (the pull-request comment) substitutes for the durable event history — the runner-local JSONL is the append target and its uploaded copy the sealed snapshot, while the projection surface belongs to the Snapshot sink of `spx/21-infrastructure.enabler/43-github-ci.enabler/21-snapshot-adapter.enabler` ([audit])
- NEVER: `vi.mock()`, `jest.mock()`, or `memfs` substitutes for the runner filesystem in the backend's tests — tests inject a controlled `StateStoreFileSystem` and exercise the real backend code paths ([audit])
