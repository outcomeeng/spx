# Issues: Runtime Config

## Tracked Follow-Ups

- `src/domains/agent-environment/runtime-config.ts`: managed runtime config state writes the absolute `productDir` into generated Codex and Claude Code runtime files. Decide whether this is intentional persisted runtime state or whether generated local runtime config files need explicit ignore/documentation guidance before broadening runtime config reconciliation beyond local agent execution.
- `src/domains/agent-environment/runtime-config.ts`: hermetic `stateDir` is caller-owned and not validated inside runtime config reconciliation. Before audit execution accepts a configured or CLI-derived state directory, `spx/36-audit.enabler/65-auditor-execution.enabler/` must validate that the resolved runtime state directory is contained by the branch-scoped audit run state boundary. Before review execution accepts a configured or CLI-derived state directory, `spx/46-reviewing.enabler/32-hermetic-review-execution.enabler/` must validate that the resolved runtime state directory is contained by the hermetic review execution state boundary. Both caller-boundary packets must add evidence before they invoke `HERMETIC_EXECUTION` from user-visible command paths.
- `src/domains/agent-environment/runtime-config.ts`: Claude Code JSON reconciliation does not define key position as a product invariant. When a file lacks the managed `spx` key, reconciliation appends the key after existing user-authored keys; when the key already exists, repeated reconciliation preserves its existing insertion position and byte-stable idempotency. Revisit only if a future instruction or review surface requires `spx` to appear first in generated JSON.
- `src/domains/agent-environment/runtime-config.ts`: Codex TOML managed-table replacement is line-oriented after parse validation. It preserves owned table boundaries and trailing separators, but does not parse TOML lexical scopes such as multiline string bodies while scanning for the next table header. Revisit with a token-aware TOML edit primitive before supporting managed sections adjacent to arbitrary multiline TOML content.
- `src/domains/agent-environment/runtime-config.ts`: `writeRuntimeConfigFile` reports directory-creation and file-write failures through the same diagnostic prefix. Split `mkdir` and `writeFile` failure messages if runtime config write diagnostics need to distinguish parent-directory setup failures from file-byte write failures.
- `spx/33-agent-environment.enabler/32-runtime-config.enabler/tests/runtime-config.scenario.l1.test.ts`: `FailingWriteFileSystem` exercises create-path write failures by returning ENOENT from `readFile`. Add a separate update-path fake if rollback diagnostics need additional evidence for pre-existing files beyond `RuntimeConfigMemoryFileSystem`.
- `spx/33-agent-environment.enabler/32-runtime-config.enabler/tests/runtime-config.scenario.l1.test.ts`: `RuntimeConfigMemoryFileSystem` uses positional optional constructor parameters for write, read, and rollback failures. Replace with an options object before adding more failure modes.
- `src/domains/agent-environment/runtime-config.ts`: hermetic runtime paths intentionally include both the runtime namespace directory and the runtime's own relative config path, for example `agent-environment/runtime-config/codex/.codex/config.toml`. Revisit only if a future runtime owns a relative path that makes the duplication user-visible or ambiguous.
