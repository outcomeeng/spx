# Issues: Runtime Config

## Tracked Follow-Ups

- `src/domains/agent-environment/runtime-config.ts`: managed configured-agent config state intentionally writes the absolute `productDir` into generated Codex and Claude Code config files. The invoking-agent outputs are local agent-tool state: `.claude/settings.local.json` is covered by `.claude/*`, and `.codex/` is ignored. Hermetic outputs must stay under their owning ignored local-state roots (`.spx/branch/{branch-slug}/audit/`, `.spx/branch/{branch-slug}/review/`, or another state-store scope explicitly added by the owning packet) before any user-visible command invokes reconciliation.
- `src/domains/agent-environment/runtime-config.ts`: hermetic `stateDir` is caller-owned and not validated inside configured-agent config reconciliation. The audit and review execution paths that once had to validate containment of the resolved configured-agent state directory collapsed into the journal channel (`spx/34-verification.enabler/21-journal.enabler/`) and were removed; if a future user-visible command derives a hermetic `stateDir` before invoking `HERMETIC_EXECUTION`, it must validate that the directory is contained by an ignored local-state boundary.
- `src/domains/agent-environment/runtime-config.ts`: Claude Code JSON reconciliation does not define key position as a product invariant. When a file lacks the managed `spx` key, reconciliation appends the key after existing user-authored keys; when the key already exists, repeated reconciliation preserves its existing insertion position and byte-stable idempotency. Revisit only if future instruction or review output requires `spx` to appear first in generated JSON.
- `src/domains/agent-environment/runtime-config.ts`: Codex TOML managed-table replacement is line-oriented after parse validation. It preserves owned table boundaries and trailing separators, but does not parse TOML lexical scopes such as multiline string bodies while scanning for the next table header. Revisit with a token-aware TOML edit primitive before supporting managed sections adjacent to arbitrary multiline TOML content.
- `src/domains/agent-environment/runtime-config.ts`: `writeRuntimeConfigFile` reports directory-creation and file-write failures through the same diagnostic prefix. Split `mkdir` and `writeFile` failure messages if configured-agent config write diagnostics need to distinguish parent-directory setup failures from file-byte write failures.
- `spx/33-agent-environment.enabler/32-runtime-config.enabler/tests/runtime-config.scenario.l1.test.ts`: `FailingWriteFileSystem` exercises create-path write failures by returning ENOENT from `readFile`. Add a separate update-path fake if rollback diagnostics need additional evidence for pre-existing files beyond `RuntimeConfigMemoryFileSystem`.
- `spx/33-agent-environment.enabler/32-runtime-config.enabler/tests/runtime-config.scenario.l1.test.ts`: `RuntimeConfigMemoryFileSystem` uses positional optional constructor parameters for write, read, and rollback failures. Replace with an options object before adding more failure modes.
- `src/domains/agent-environment/runtime-config.ts`: hermetic configured-agent paths intentionally include both the configured-agent namespace directory and the configured agent's own relative config path, for example `harness-environment/runtime-config/codex/.codex/config.toml`. Revisit only if a future configured agent owns a relative path that makes the duplication user-visible or ambiguous.
