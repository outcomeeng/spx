# Open Issues

## Session IDs use local time and break the lexicographic-ordering invariant around DST

`generateSessionId` and `parseSessionId` in `src/domains/session/timestamp.ts` build and parse session IDs from local-time components — `getFullYear`/`getMonth`/`getDate`/`getHours`/`getMinutes`/`getSeconds` and the local `new Date(year, month, day, hours, minutes, seconds)` constructor. Around a daylight-saving fall-back transition one local wall-clock hour occurs twice, so the ID is not a monotonic function of the instant and the round-trip is not epoch-lossless.

**Evidence:** The property test `tests/session-identity.property.l1.test.ts` (`GIVEN any two valid Dates WHEN IDs compared THEN lexicographic order matches chronological order`) failed on the fast-check counterexample `[new Date("2087-10-26T01:00:00.000Z"), new Date("1999-12-31T23:00:00.000Z")]` (seed `-642661936`) when run in a CET/CEST timezone: `parseSessionId(generateSessionId(left))?.getTime()` differed from `truncateToSecond(left)` by exactly `3600000` ms (one hour). 2087-10-26 is the European DST fall-back day (last Sunday of October). The suite passes in UTC (no DST), which is why CI — and most local seeds, which do not land on a DST boundary — never surface it.

**Impact:** Violates the invariant declared by [`spx/36-session.enabler/21-timestamp-format.adr.md`](../21-timestamp-format.adr.md) — "Lexicographic ordering of session IDs matches chronological ordering." Two distinct instants in the DST-repeated hour collapse to the same ID, and a later instant can sort equal-to or earlier-than an earlier one. Session ordering (pickup priority, list order) is unreliable for sessions created during the ~1-hour annual fall-back window in DST timezones.

**Resolution condition:** Generate and parse session IDs in UTC — `getUTCFullYear`/`getUTCMonth`/`getUTCDate`/`getUTCHours`/`getUTCMinutes`/`getUTCSeconds` and `new Date(Date.UTC(...))` — making the wall-clock representation monotonic and the round-trip epoch-lossless. This also requires updating the timezone-dependent assertions in `tests/session-identity.scenario.l1.test.ts` and `tests/session-identity.compliance.l1.test.ts` (the ~5 `it`-blocks that construct `new Date(year, month, day, ...)` locally and assert local `get*` getters) to use `Date.UTC(...)` construction and `getUTC*` getters so they hold independently of the machine timezone. Run through `/spec-tree:applying` on `spx/36-session.enabler/32-session-identity.enabler` with the test-audit and code-audit gates. Update the `(local time)` notes in the `timestamp.ts` JSDoc to UTC as part of the fix.
