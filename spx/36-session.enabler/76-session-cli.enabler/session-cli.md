# Session CLI

PROVIDES Commander.js bindings for all session subcommands with variadic ID parsing (single-ID `reconcile` excepted), per-ID result reporting, non-zero exit on any failure, and parseable `<HANDOFF_ID>`/`<PICKUP_ID>` tag emission
SO THAT agents and automation tools
CAN invoke session operations from the command line with predictable output and exit codes

## Assertions

### Scenarios

- Given three sessions in `todo`, when `spx session archive` is invoked with the three IDs, then all three move to `archive/` and the result reports them archived ([test](tests/session-cli.scenario.l1.test.ts))
- Given a session whose frontmatter omits structured fields or carries keys outside the shape, when `spx session archive` is invoked, then the session moves to `archive/` per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/session-cli.scenario.l1.test.ts))
- Given three session IDs, when `spx session delete` is invoked with all three, then no file with any of the three IDs remains in `todo/`, `doing/`, or `archive/` ([test](tests/session-cli.scenario.l1.test.ts))
- Given two session IDs, when `spx session show` is invoked with both, then both session bodies appear in the result, each labeled with its priority ([test](tests/session-cli.scenario.l1.test.ts))
- Given two sessions claimed in `doing`, when `spx session release` is invoked with both IDs, then both move from `doing/` to `todo/` ([test](tests/session-cli.scenario.l1.test.ts))
- Given two sessions in `todo`, when `spx session pickup` is invoked with both IDs, then both move to `doing/` and the result carries a `<PICKUP_ID>` for each ([test](tests/session-cli.scenario.l1.test.ts))
- Given one session ID in `doing` and one invalid session ID, when `spx session release` is invoked with both, then the valid session moves to `todo/` and the operation fails, naming the invalid ID ([test](tests/session-cli.scenario.l1.test.ts))
- Given one valid session ID and one ID that fails its operation, when `spx session archive` is invoked with both, then the valid session is archived and the operation fails, naming the failing ID ([test](tests/session-cli.scenario.l1.test.ts))
- Given a JSON header that omits `goal`, when `spx session handoff` is executed through `node bin/spx.js`, then no file is written, a diagnostic line naming `SessionInvalidGoalError` is written to stderr, and the process exits non-zero ([test](tests/session-cli.compliance.l2.test.ts))
- Given stdin opening with the YAML-frontmatter delimiter `---`, when `spx session handoff` is executed through `node bin/spx.js`, then no file is written, a diagnostic line naming `SessionLegacyFrontmatterInputError` is written to stderr, and the process exits non-zero per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/session-cli.compliance.l2.test.ts))
- Given a malformed JSON header (opens with `{` but is not a valid JSON object), when `spx session handoff` is executed through `node bin/spx.js`, then no file is written, a diagnostic line naming `SessionInvalidJsonHeaderError` is written to stderr, and the process exits non-zero per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/session-cli.compliance.l2.test.ts))
- Given a JSON header followed by body bytes with leading and trailing whitespace, when `spx session handoff` is executed through `node bin/spx.js`, then the written session file preserves those body bytes after the JSON-prefix separator per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/session-cli.compliance.l2.test.ts))
- Given a permitted handoff base and a JSON header carrying a `git_ref` work-branch ref that exists on `origin`, when `spx session handoff` is executed through `node bin/spx.js`, then the written session file's `git_ref` equals that work-branch ref, stdout carries the `<HANDOFF_ID>` and `<SESSION_FILE>` tags, and the process exits 0 per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/session-cli.compliance.l2.test.ts))
- Given a JSON header carrying a `git_ref` work-branch ref that does not exist on `origin`, when `spx session handoff` is executed through `node bin/spx.js`, then no file is written, a diagnostic line naming `SessionWorkBranchNotOnOriginError` is written to stderr, and the process exits non-zero per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/session-cli.compliance.l2.test.ts))
- Given a single session ID, when `spx session archive` is invoked with that ID, then one archive operation is performed and its result is reported ([test](tests/session-cli.scenario.l1.test.ts))
- Given a single session ID, when `spx session delete` is invoked with that ID, then one delete operation is performed and its result is reported ([test](tests/session-cli.scenario.l1.test.ts))
- Given a single session ID, when `spx session show` is invoked with that ID, then one show operation is performed and its result is reported ([test](tests/session-cli.scenario.l1.test.ts))
- Given a single session ID, when `spx session pickup` is invoked with that ID, then one pickup operation is performed and its result is reported ([test](tests/session-cli.scenario.l1.test.ts))
- Given a single session ID, when `spx session release` is invoked with that ID, then one release operation is performed and its result is reported ([test](tests/session-cli.scenario.l1.test.ts))
- Given a non-main checkout, when `spx session handoff` is executed through `node bin/spx.js`, then no file is written, the process exits non-zero, and the rendered `SessionHandoffBaseError` checklist — naming a base prerequisite — is written to stderr per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/session-cli.compliance.l2.test.ts))
- Given a permitted main-checkout base, when `spx session handoff` is executed through `node bin/spx.js`, then the session is written, stdout carries the `<HANDOFF_ID>` and `<SESSION_FILE>` tags, no checklist is written to stderr, and the process exits 0 per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/session-cli.compliance.l2.test.ts))
- Given a permitted non-main-checkout base — a clean worktree detached at the `origin/<default>` tip — when `spx session handoff` is executed through `node bin/spx.js`, then the session is written, stdout carries the `<HANDOFF_ID>` and `<SESSION_FILE>` tags, no checklist is written to stderr, and the process exits 0 per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/session-cli.compliance.l2.test.ts))
- Given a non-main checkout in a bare-repository pool, when `spx session handoff` is executed through `node bin/spx.js`, then the rendered checklist's main-checkout fact line names the repository-named worktree path when the pool designates a main checkout, and renders unresolved when the pool designates none — whether because no `origin` remote names a repository or because the named worktree is absent — per [`spx/15-worktree-management.pdr.md`](../../15-worktree-management.pdr.md) ([test](tests/session-cli.compliance.l2.test.ts))

### Mappings

- The list color decision maps each combination of `(stdout is a TTY, NO_COLOR is present, color flag ∈ {--color, --no-color, unset})` to enabled or disabled: enabled when stdout is a TTY with neither `NO_COLOR` nor `--no-color`, disabled when stdout is not a TTY or `NO_COLOR` is present, with `--color` forcing enabled and `--no-color` forcing disabled over both the TTY state and `NO_COLOR` ([test](tests/list-text.mapping.l1.test.ts))

### Properties

- For every input list of session IDs `[id_1, ..., id_n]` produced by `arbitraryBatchInputs(n, validCount)` (with `n >= 1` and `0 <= validCount <= n`), running the variadic subcommand through `node bin/spx.js` produces exactly `validCount` success outputs and `n - validCount` error lines on stderr ([test](tests/session-cli.property.l2.test.ts))
- For every input list of session IDs `[id_1, ..., id_n]`, the order of per-ID result outputs in stdout and stderr matches the order of `id_1, ..., id_n` on the command line ([test](tests/session-cli.property.l2.test.ts))
- For every session list and every terminal width at or above the formatter's documented minimum, every line the text formatter renders has an escape-stripped display width no greater than that width ([test](tests/list-text.property.l1.test.ts))
- For every non-empty session list, the text formatter renders no ANSI escape sequence when color is disabled and renders ANSI styling escapes when color is enabled ([test](tests/list-text.property.l1.test.ts))
- For every `HandoffBaseChecklist`, `renderHandoffBaseChecklist` emits a header line naming `SessionHandoffBaseError`, a git-facts section with exactly five resolved-fact lines, a checks-status section, and one numbered line per base prerequisite per [`spx/36-session.enabler/76-session-cli.enabler/21-handoff-base-rendering.adr.md`](21-handoff-base-rendering.adr.md) ([test](tests/handoff-base-render.property.l1.test.ts))
- For every `HandoffBaseChecklist`, each resolved-fact line round-trips its field — an absent value renders the unresolved sentinel and never the literal `origin/<default>` — and each prerequisite line carries `[PASS]` with no remedy, or `[FAIL]` with its remedy, per [`spx/36-session.enabler/76-session-cli.enabler/21-handoff-base-rendering.adr.md`](21-handoff-base-rendering.adr.md) ([test](tests/handoff-base-render.property.l1.test.ts))
- For every dirty `HandoffBaseChecklist`, the first diagnostic line states that the checkout is dirty and cannot hand off, and the clean-working-tree prerequisite line is numbered `[FAIL]` with no main-checkout alternative per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/handoff-base-render.property.l1.test.ts))
- For every `HandoffBaseChecklist`, no remedy `renderHandoffBaseChecklist` emits suggests `git stash` — the closed remedy set directs the agent to commit, detach, or hand off from the main checkout, never to hide work outside the shared queue per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/handoff-base-render.property.l1.test.ts))

### Compliance

- ALWAYS: every variadic subcommand processes all provided IDs even when an earlier ID fails — partial failures do not halt processing ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: per-ID success and failure outputs identify the ID so the caller can map results back to inputs ([test](tests/session-cli.property.l2.test.ts))
- ALWAYS: the process exits with a non-zero code when any ID fails — partial success is still a failure exit code ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: diagnostic lines for `SessionInvalidContentError`, `SessionInvalidGoalError`, `SessionInvalidNextStepError`, `SessionWorkBranchNotOnOriginError`, `SessionLegacyFrontmatterInputError`, and `SessionInvalidJsonHeaderError` per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) include the error name and the offending session ID (when applicable) ([test](tests/session-cli.compliance.l2.test.ts))
- NEVER: a variadic subcommand silently drops IDs beyond the first — every argument is processed ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: the rendered `SessionHandoffBaseError` checklist written to stderr carries the git facts the handler collected — the observed HEAD commit SHA, the current worktree path, and the resolved default branch and `origin/<default>` tip when set, each absent value stated as unresolved, and a dirty-checkout headline with the clean-working-tree prerequisite marked `[FAIL]` when the handler detects a dirty tree — per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/session-cli.compliance.l2.test.ts))
- NEVER: the rendered `SessionHandoffBaseError` diagnostic written to stderr suggests `git stash` as a remedy — the descriptor surfaces only the formatter's closed remedy set per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/session-cli.compliance.l2.test.ts))
- NEVER: the rendered `SessionHandoffBaseError` diagnostic written to stderr renders the literal placeholder `origin/<default>` for an unresolved base — an unresolved value is stated as unresolved per [`spx/36-session.enabler/11-session-frontmatter.pdr.md`](../11-session-frontmatter.pdr.md) ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: `spx session list`, `todo`, `show`, `delete`, `pickup`, `reconcile`, `release`, `archive`, and `prune`, each run outside a git repository, write the non-git-repo diagnostic to stderr per [`spx/15-worktree-management.pdr.md`](../../15-worktree-management.pdr.md); `handoff` instead refuses a non-git base with `SessionHandoffBaseError` and surfaces no diagnostic ([test](tests/session-cli.compliance.l2.test.ts))
- NEVER: the non-git-repo diagnostic a session subcommand writes to stderr claims that sessions will be created — the diagnostic names session-storage resolution so it reads correctly for the read, claim, and retention subcommands that write it; `handoff` writes no non-git diagnostic ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: `spx session list --json` executed through `node bin/spx.js` writes parseable JSON to stdout keyed by status directory with each session a flat record per [`spx/36-session.enabler/43-session-store.enabler/session-store.md`](../43-session-store.enabler/session-store.md), and exits 0 ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: `spx session show <id> --json` executed through `node bin/spx.js` writes the session's flat record as parseable JSON to stdout — a bare record object for a single id, a JSON array in supplied order for several — per [`spx/36-session.enabler/43-session-store.enabler/session-store.md`](../43-session-store.enabler/session-store.md), and exits 0 ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: `spx session show --json` invoked with an unresolvable id processes every supplied id, writes the not-found diagnostic naming that id to stderr, and exits non-zero, writing no record JSON to stdout — failure under JSON output stays on stderr, mirroring the text-mode variadic contract ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: `spx session show <id>` without `--json` writes the metadata header and session body to stdout and exits 0, leaving the default text view unchanged by the JSON mode ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: `spx session reconcile <id>` executed through `node bin/spx.js` writes the session's reconciliation findings — one verdict per recorded reference per [`spx/36-session.enabler/54-session-reconciliation.enabler/session-reconciliation.md`](../54-session-reconciliation.enabler/session-reconciliation.md) — as parseable JSON to stdout and exits 0 ([test](tests/session-cli.compliance.l2.test.ts))
- NEVER: `spx session reconcile` accepts more than one session ID operand — the per-session reconciliation report is the declared exception to variadic ID parsing ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: `spx session list --fields id,priority,goal,next_step,git_ref` and `spx session todo --fields id,priority,goal,next_step,git_ref` executed through `node bin/spx.js` write parseable JSON whose every session record carries exactly those five fields, and exit 0 ([test](tests/session-cli.compliance.l2.test.ts))
- NEVER: a `--fields` value that names no valid field — a token outside the session record field set, or a value naming no field at all (empty or only separators) — produces JSON on stdout; `node bin/spx.js` writes a diagnostic naming the offending value and the valid field set to stderr and exits non-zero ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: when stdout is not a TTY, `spx session list` and `spx session todo` emit plain text carrying no ANSI escape sequence, so piped and redirected output stays parse-safe ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: `--no-color` forces plain output and `--color` forces styled output even when `NO_COLOR` is present, so an explicit color flag overrides the environment ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: `spx session list --json`, `spx session list --fields …`, and the same `spx session todo` projections emit neither color nor truncation regardless of TTY state or color flag, leaving the JSON projection of [`spx/36-session.enabler/43-session-store.enabler/session-store.md`](../43-session-store.enabler/session-store.md) unchanged ([test](tests/session-cli.compliance.l2.test.ts))
- ALWAYS: priority styling derives its colors from the `SESSION_PRIORITY` values rather than hardcoded priority strings, so a new priority value cannot ship with an unstyled badge ([audit])
- NEVER: the text formatter reads process state — `process.stdout.isTTY`, `process.env`, or `NO_COLOR` — the color decision and terminal width are resolved in the descriptor and passed to the formatter as parameters per [`spx/14-cli-composition.adr.md`](../../14-cli-composition.adr.md) ([audit])
