# Node Status Architecture

The node-status library is one module at `src/lib/node-status/` whose public surface (`index.ts`) exposes a pure status-file parser, a lifecycle classifier over spec structure plus verification outcomes, an exact-node EXCLUDE membership reader, an evidence-provider factory, and the `spx spec status --update` orchestration entry point with the verification resolver type it accepts; the writer is an internal module the orchestration alone reaches, never a public export. The read path joins `spx spec status` through a `SpecTreeEvidenceProvider` the factory builds by closing over `productDir`, resolving each node's directory from the node entry's `id`/`ref.path` to locate its `spx.status.json`, and returning the lifecycle state derived from persisted verification outcomes when present and `undefined` when absent so the spec-tree library's live `deriveState` governs. The `--update` orchestration obtains per-node verification outcomes through dependency-injected resolvers supplied at the command edge per `spx/31-spec-domain.enabler/54-spec-cli-commands.enabler/21-status-testing-delegation.adr.md`, and folds them over the node's committed outcomes: the resolver reports an outcome for an evidence reference a recorded run covers, and reports a reference no recorded run covers as `not-run` rather than leaving it unresolved, so only a covered-but-stale reference is carried forward from the node's committed `spx.status.json` through the library's own reader — a fold thus neither discards a claim a run still supports nor preserves one no run produces, which every regeneration would reproduce and none could refute; the library composes no test runner, lifecycle values come from `SPEC_TREE_NODE_STATE`, EXCLUDE membership comes from the node-status EXCLUDE membership reader, tracked `spx/` paths resolve under `productDir` per `spx/15-worktree-management.pdr.md`, and the file contract is governed by `spx/31-spec-domain.enabler/21-node-status.enabler/15-status-file-contract.pdr.md`. EXCLUDE membership matches a listed node id or the node `ref.path` after stripping the leading `spx/` root segment; membership does not cascade from an excluded ancestor to descendant nodes. The `--update` orchestration writes status only for git-tracked node directories — a node directory participates only when git tracks at least one file under it: `updateNodeStatus` accepts a dependency-injected git runner, resolves the tracked-path set through it, and skips any node directory git does not track, so a stale, untracked, node-shaped directory neither receives a written `spx.status.json` nor survives the stale-file sweep. The boundary is the node directory, not the individual file, so a tracked node's own not-yet-tracked evidence stays recorded in full. The tracked-path query and predicate are node-status's own git-plumbing helpers, governed by this decision as the consumer that defines the boundary. Node visibility stays structural: `spx spec status` without `--update` applies no tracked filter, so a newly authored, not-yet-tracked node still appears — only the `--update` write set is restricted to git-tracked directories.

## Rationale

A single library module mirrors the established `src/lib/spec-tree/` and `src/lib/file-inclusion/` boundaries, so consumers import one surface rather than reaching into file IO, EXCLUDE parsing, verification outcome parsing, or classification internals. Keeping the writer internal — reachable only through the `--update` orchestration — makes the file contract's single-writer rule a structural property of the boundary: there is no exported writer to call from another path. Splitting the concern into a status-file parser, a pure classifier, a thin reader, an internal writer, and a provider factory isolates the expensive inputs — test, eval, and audit execution outcomes — behind dependency-injected resolvers, so lifecycle derivation remains `l1`-verifiable without running a real suite.

Reusing the spec-tree `SpecTreeEvidenceProvider.stateForNode` hook (which supplies only the node source entry and in-memory evidence, no filesystem path, hence the factory closing over `productDir` and mapping via `id`/`ref.path`) means a node with a persisted file yields its derived recorded state and a node without one yields `undefined`, which `deriveState` already handles as live derivation. Owning the EXCLUDE membership reader inside node-status keeps lifecycle classification independent from file-inclusion scope decisions, while `SPEC_TREE_NODE_STATE` single-sources lifecycle vocabulary. A status file containing top-level lifecycle state alone is rejected because it loses which verification references produced the result. Placing the module inside `src/lib/spec-tree/` (backend-neutral, owns neither EXCLUDE nor test execution), exposing the writer publicly (defeats the single-writer rule), a bespoke command-side read path (duplicates the live-derivation fallback and couples the command to file layout), and using the file-inclusion scope resolver for lifecycle EXCLUDE membership are rejected.

The write set is git-tracked membership, not the broader git-visible scope `spx/17-file-inclusion.enabler/file-inclusion.md` provides: git-visible admits an untracked-but-unignored path, so a stale node-shaped directory holding only leftover untracked files would remain visible and keep receiving status. Tracked membership — git tracks at least one file under the directory — is the boundary the status contract names, so it is a thin git probe over the injected runner rather than a file-inclusion dependency. The filter is the node directory, not the walked path: `updateNodeStatus` walks the whole spec tree and skips a node whose directory git does not track, rather than filtering the filesystem source per entry. A per-entry filter would also drop a tracked node's not-yet-staged evidence file, so `--update` would omit a current verification that CI records once the file is committed; filtering at the node-directory boundary keeps that evidence and matches CI. The git runner is injected into `updateNodeStatus` with a production default, mirroring how the orchestration already constructs its EXCLUDE reader and filesystem source from `productDir`; outside a git repository the query yields no scoping and every node directory is written, so non-git fixtures see every node.

## Invariants

- Classification is total and deterministic: every combination of spec structure, EXCLUDE-membership, and persisted verification outcomes maps to exactly one `SPEC_TREE_NODE_STATE` value, and identical inputs always map to the same value.
- EXCLUDE membership is exact: a listed node id or node ref path matches that node only, and descendants require their own listed entry.
- Round-trip fidelity: reading a node directory the writer has populated yields the same verification outcomes and derived lifecycle state the writer recorded.
- Read purity: resolving a node's persisted state performs no filesystem writes and runs no tests.
- Fold preservation: an evidence reference whose recorded evidence is stale retains the outcome the node's committed status file records, while a reference no recorded run covers is `not-run` regardless of what the committed file records — a preserved outcome no run produces would be reproduced by every regeneration and so could never be refuted.
- Tracked-only write set: the `--update` node set contains a node-shaped directory only when git tracks at least one file under it, so a node-shaped directory with no git-tracked file is absent from both the written status set and the status files the stale-file sweep retains. The boundary is the directory: tracked directories record evidence in full, including an evidence file not yet individually tracked.

## Verification

### Testing

- ALWAYS: parsing `spx.status.json` accepts only schema version 1 files whose verification mechanisms, overall values, evidence references, and per-reference outcomes conform to `spx/31-spec-domain.enabler/21-node-status.enabler/15-status-file-contract.pdr.md` ([conformance])
- ALWAYS: lifecycle classification maps persisted verification outcomes to `SPEC_TREE_NODE_STATE` values deterministically for all valid mechanism rollups ([mapping])
- ALWAYS: the read path returns `undefined` when `spx.status.json` is absent and performs no writes or verification execution when it is present ([compliance])
- ALWAYS: `spx spec status --update` writes `spx.status.json` only into git-tracked node directories; a node-shaped directory with no git-tracked file under it receives no written status file and any pre-existing `spx.status.json` there is removed by the stale-file sweep ([compliance])
- ALWAYS: `spx spec status --update` records a git-tracked node directory's evidence in full, including an evidence file under it that is not yet individually tracked ([compliance])
- ALWAYS: the tracked-path query reads git-tracked paths under `productDir` through the injected git runner and yields no tracked set outside a git repository ([mapping])
- ALWAYS: the tracked-path predicate admits a path exactly when it is a git-tracked file or an ancestor directory of one, and admits every path when no tracked set is available ([property])

### Audit

- ALWAYS: expose the status-file parser, lifecycle classifier, reader, evidence-provider factory, and `spx spec status --update` orchestration with its verification resolver type through a single `src/lib/node-status/index.ts` public surface — consumers import the surface, not internals ([audit])
- ALWAYS: keep the writer an internal module the `--update` orchestration alone reaches ([audit])
- ALWAYS: express lifecycle values through `SPEC_TREE_NODE_STATE` — node-status declares no parallel state strings ([audit])
- ALWAYS: resolve EXCLUDE membership through the node-status EXCLUDE membership reader exported from `src/lib/node-status/` ([audit])
- ALWAYS: build the evidence provider in a factory that closes over `productDir` and resolves each node's directory from the node entry's `id`/`ref.path`, passing the provider to `readSpecTree` and returning the derived persisted state when present and `undefined` when absent ([audit])
- ALWAYS: accept per-node verification outcomes through dependency-injected resolver parameters — enables `l1` verification of classification without executing a real suite ([audit])
- ALWAYS: the `--update` orchestration folds resolved outcomes over the node's committed outcomes, reading the committed file through the library's own status reader, so an evidence reference the resolver leaves unresolved keeps its committed outcome ([audit])
- NEVER: the `--update` orchestration keeps a committed outcome for an evidence reference no recorded run covers — the resolver distinguishes a covered-but-stale reference, whose committed outcome is carried forward, from an uncovered one, which reads `not-run` ([audit])
- ALWAYS: accept `productDir` from the caller and resolve tracked `spx/` paths under it per `spx/15-worktree-management.pdr.md` ([audit])
- ALWAYS: the `--update` orchestration accepts a dependency-injected git runner (defaulting to the production runner), resolves the tracked-path set through it, and skips a node directory git does not track — filtering at the node-directory boundary rather than per walked filesystem entry, so a tracked node's not-yet-tracked evidence stays recorded ([audit])
- ALWAYS: the tracked-path query and predicate are the node-status boundary's own git-plumbing helpers, governed here as the consumer that defines the tracked boundary; they read git state only through the injected git runner (`GitDependencies`), never a direct process call ([audit])
- NEVER: `vi.mock()`, `jest.mock()`, or any module-interception mechanism for node-status dependencies — classification, reader, writer, and the tracked-path predicate are exercised against real temporary git repositories or an injected controlled git runner ([audit])
- NEVER: read git state in the node-status library through a direct `child_process` call or any path other than the injected git runner ([audit])
- NEVER: classify or write status for a node-shaped directory that git does not track ([audit])
- NEVER: export the writer from the public surface or invoke it outside the `--update` orchestration ([audit])
- NEVER: run tests, evals, audits, or write any file on the read path — reading persisted state is pure ([audit])
- NEVER: declare the four lifecycle string literals anywhere outside `SPEC_TREE_NODE_STATE` ([audit])
- NEVER: read EXCLUDE membership through file-inclusion scope resolution — lifecycle EXCLUDE membership belongs to node-status, while file-inclusion governs walked file scope ([audit])
- NEVER: the node-status library composes a language-specific test runner — outcomes arrive through the injected resolver ([audit])
