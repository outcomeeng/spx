# CLI Architecture

The SPX CLI partitions source between CLI command-interface concerns under `src/interfaces/cli/` and shared process or diagnostic utilities under `src/lib/`. CLI-interface modules adapt an invocation host into command inputs and injected capabilities: Commander wiring, argv/options, stdin/stdout/stderr, process exit, terminal state, and process-environment reads that select an environment-bound capability. They do not own backend transport mechanics merely because a backend uses a process, network, or API client; those mechanics belong to the backend's product concern behind its injected boundary. Process-lifecycle handling installs once at CLI entry (`installLifecycle()` as the first executable statement in `src/cli.ts`), registers every asynchronously spawned child in a module-scoped registry exposed as a `ProcessRunner`-conformant `lifecycleProcessRunner`, exposes a managed subprocess helper that owns parent-piped stdio for long-running children, and forwards SIGINT, SIGTERM, EPIPE, and uncaught-exception events to all registered children with the conventional exit codes (130, 143, 0, 1) before the parent exits. A foreground exec-handoff is the one spawn category outside this tracking model: a non-registering foreground runner spawns a terminal-owning child with inherited stdio without adding it to the registry, and the parent ignores SIGINT and SIGTERM for the child's lifetime so the foreground child receives them directly, then exits with the child's status. User-supplied argument sanitization is a shared pure diagnostic utility under `src/lib/` because CLI transports and hook transports both echo external input in terminal diagnostics. Escaping is a property of the composed value rather than a habit of each write site: terminal-destined text is composed through the `src/lib/terminal-text/` primitive, which escapes every externally-originated segment — subprocess output, filesystem paths, file content, environment variables, argv, caught-error messages, and API responses — while preserving the bytes of product-authored segments, so ANSI styling and line structure survive. This governs every module under `src/interfaces/cli/`, `src/interfaces/hooks/`, and `src/lib/` that participates in terminal-boundary diagnostics or process lifecycle; it refines `spx/15-worktree-management.pdr.md` and `spx/19-language-registration.adr.md` without contradicting them.

## Rationale

Layering separates invocation adaptation from product capability construction: command-interface concerns — subcommand dispatch, option wiring, package-script invariants, standard-stream plumbing, exit-code reporting, and invocation-host environment reads — operate at the Commander.js boundary and belong under `src/interfaces/cli/`, while shared diagnostic and process utilities belong under `src/lib/` when multiple interfaces consume them. A CLI descriptor may instantiate an imported production backend capability so the command receives an explicit dependency, but the descriptor does not decide that backend's API shape, authentication, pagination, subprocess command, or write semantics. Argument sanitization is deliberately shared because `spx validation <unknown>` and `spx hook run <unknown>` both echo external operands to stderr and require identical byte-safety behavior. Process-lifecycle concerns — signal forwarding, EPIPE handling, child tracking — operate on Node process state regardless of entrypoint and would be reused unchanged by a daemon, worker, or REPL. A module-scoped registry installed once, paired with a `ProcessRunner` adapter, lets every domain runner receive an injected runner (production wires `lifecycleProcessRunner`, tests inject any conforming object) without threading a registry argument through every spawn site or forcing test-runner module resets between cases. The exit-code mapping follows POSIX — 128 + signal number for signal termination, 0 for downstream-closed pipes (matching `head` / `tee`), non-zero for genuine failures — so `set -o pipefail` and CI exit-code checks behave as operators expect. Handlers install in the order EPIPE → uncaughtException → SIGTERM → SIGINT with a `cleanupOnce` guard and per-child `killed` checks so the first triggered path engages and the SIGINT-mid-write-to-a-closed-pipe race resolves to a single cleanup.

Escaping at the write boundary is rejected: a rendered report is one composite string whose labels are product-authored and whose readings are external, so a writer that escapes everything destroys the product's own styling and line structure, and a writer that escapes nothing is the discipline the primitive replaces. The trust decision belongs where a value is embedded, which is the only place both provenances are still distinguishable. Escaping at each call site is rejected for the reason the defect record shows: the property is then re-proved by hand once per site and is discovered missing only when someone notices, so one boundary closed leaves every sibling boundary open.

Rejected: a single `src/cli/` directory (conflates what runs with what wraps the run, forcing non-CLI entrypoints to import a misleadingly-named module or duplicate the lifecycle logic); inline lifecycle in `src/cli.ts` (spreads handler installation, registry state, and the runner adapter across one large file that resists unit testing at the registry, handler, and adapter levels); a process group with `detached: true` (changes signal semantics for children that internally spawn, such as `tsc -w`, and complicates exit-code propagation); and per-call-site cleanup (duplicates the registry pattern at every site and makes consistent SIGINT handling impossible).

## Invariants

- For every successful CLI invocation, the count of registered children at process exit is zero.
- For every signal-terminated invocation (SIGINT or SIGTERM), every child observed in the registry receives the corresponding signal exactly once.
- For every EPIPE on `process.stdout`, the next observable process state is exit code 0 with no `uncaughtException` text on `process.stderr`.
- For every spawn site that tracks a long-running child in production, the injected runner is the shared `lifecycleProcessRunner`; a foreground exec-handoff is the sole exception, consuming the non-registering foreground runner.
- For every long-running child process launched through the managed subprocess helper, the child receives parent-owned piped stdio rather than inherited outer stdout/stderr descriptors.
- For every foreground exec-handoff, the child is spawned with inherited stdio and is not added to the lifecycle registry, and the parent ignores SIGINT and SIGTERM while the child runs, then exits with the child's status.

## Verification

### Audit

- ALWAYS: `installLifecycle()` is the first executable statement in `src/cli.ts`, preceding any domain registration ([audit])
- ALWAYS: the `lifecycleProcessRunner` exported from `src/lib/process-lifecycle/` is structurally compatible with the `ProcessRunner` interface so it substitutes for any `{ spawn }` literal default ([audit])
- ALWAYS: SIGINT, SIGTERM, EPIPE, and uncaughtException handlers are installed once per process; subsequent installations are no-ops ([audit])
- ALWAYS: the lifecycle signal-to-exit-code mapping is SIGINT → 130, SIGTERM → 143, EPIPE → 0, uncaughtException → 1 ([audit])
- ALWAYS: cleanup is idempotent — invoking the SIGINT or SIGTERM handler N times kills each registered child exactly once ([audit])
- ALWAYS: CLI command-interface modules — dispatch primitives, command descriptors, and package-script invariants — live under `src/interfaces/cli/` ([audit])
- ALWAYS: CLI-interface modules adapt invocation-host state into explicit command inputs and injected capabilities; backend API mechanics, authentication, pagination, and backend subprocess command construction live with the backend concern that owns them ([audit])
- ALWAYS: shared pure terminal-diagnostic helpers consumed by multiple interface layers, including argument sanitization, live under `src/lib/` and are imported directly by their consumers without compatibility re-export shims ([audit])
- ALWAYS: a value that originated outside the product's own source — subprocess output, a filesystem path, file content, an environment variable, an argv operand, a caught-error message, or an API response — is escaped at the point it is embedded into terminal-destined text, through the `src/lib/terminal-text/` composition primitive ([audit])
- ALWAYS: product-authored segments of terminal-destined text keep their bytes through composition, so ANSI styling emitted by a product renderer and the line structure of a multi-line diagnostic survive escaping ([audit])
- NEVER: escape an externally-originated value at the process-stream write site; a composed report mixes authored and external segments, and the write site can no longer tell them apart ([audit])
- NEVER: escape a value twice by re-composing text a caller already composed ([audit])
- ALWAYS: process-lifecycle modules — registry, handlers, lifecycle runner, install entrypoint — live under `src/lib/process-lifecycle/` ([audit])
- ALWAYS: domain runners that spawn long-running subprocesses accept the runner through dependency injection and consume `lifecycleProcessRunner` as the production default ([audit])
- ALWAYS: domain runners that spawn long-running subprocesses use the managed subprocess helper instead of setting `stdio` directly at the call site ([audit])
- ALWAYS: domain runners that spawn long-running subprocesses drain the child stdout and stderr streams through the domain output adapter ([audit])
- NEVER: import `child_process.spawn` for asynchronous child processes outside `src/lib/process-lifecycle/`; synchronous `execSync` and `spawnSync` are exempt because they self-reap before parent exit ([audit])
- NEVER: set `detached: true` on subprocess spawns in production code paths; detachment changes signal semantics and contradicts the registry's tracking model ([audit])
- NEVER: set `stdio` directly at a long-running subprocess call site that consumes `ProcessRunner`; the managed subprocess helper owns that policy ([audit])
- NEVER: spawn a child via `lifecycleProcessRunner.spawn` without registering its handle in the lifecycle registry ([audit])
- ALWAYS: a foreground exec-handoff spawns its terminal-owning child through the non-registering foreground runner under `src/lib/process-lifecycle/`, setting `stdio: "inherit"` directly, and never adds it to the lifecycle registry or routes it through the managed subprocess helper ([audit])
- ALWAYS: while a foreground exec-handoff child runs, the parent ignores SIGINT and SIGTERM so the foreground child receives them directly, then exits with the child's status ([audit])
- NEVER: use `vi.mock()` or `jest.mock()` to replace the spawn primitive, the registry, or the lifecycle runner; tests inject controlled implementations through the `ProcessRunner` interface ([audit])
- NEVER: place command-interface policy under `src/lib/`, process-lifecycle code under `src/interfaces/`, or compatibility re-export shims outside a shared utility's canonical owner ([audit])
- NEVER: place a hosted backend's transport mechanics under `src/interfaces/cli/` merely because its production adapter uses a process, network client, or environment-bound credential; the CLI interface wires the adapter as an injected capability rather than owning it ([audit])
- NEVER: write to `process.stdout` or `process.stderr` without an `error` listener installed; unhandled EPIPE becomes an uncaught exception that violates the exit-code mapping ([audit])
