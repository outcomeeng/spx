# Open Issues

## Two patterns coexist for descriptor stderr warning writes

- Review: CI `spec-tree-review` on PR #90 — F-002 (standards, follow_up). [Comment](https://github.com/outcomeeng/spx/pull/90#issuecomment-4587847103).
- Evidence: Three CLI-interface descriptors emit an undecorated warning string to stderr with a trailing newline. `src/interfaces/cli/spec.ts:84-86` uses a named `writeWarning(warning)` helper that calls `console.error(warning)`. `src/interfaces/cli/config.ts:28` uses an inline `process.stderr.write(\` ${resolved.warning}\n\`)`. `src/interfaces/cli/session.ts:177` uses the same inline `process.stderr.write(\`${result.warning}\n\`)` shape. The output bytes are identical across all three, but the API and the encapsulation are not.
- Impact: A future author adding a fourth warning-emitting descriptor sees two patterns to choose from, with no documented preference. The split also blocks any future change to warning formatting (a prefix tag, a colorized stream, structured output for `--json`) from landing in one place — every site has to be edited independently.
- Resolution: Decide whether `console.error(warning)` (spec.ts model) or `process.stderr.write(\`${warning}\n\`)`(config.ts and session.ts model) is canonical for descriptor warning writes, then update the other sites and extract a shared helper into the CLI-interface layer. The shared helper home is`src/interfaces/cli/` since the operation is descriptor-scoped, not domain-scoped.

## Descriptor warning guard predicate disagrees with handler filter

- Review: CI `spec-tree-review` on PR #90 — F-003 + F-004 (consistency, debt — cyclic). [F-003](https://github.com/outcomeeng/spx/pull/90#issuecomment-4587873195), [F-004](https://github.com/outcomeeng/spx/pull/90#issuecomment-4587887795).
- Evidence: The handoff handler at `src/commands/session/handoff.ts:139` uses a truthy guard — `warning ? { output, warning } : { output }` — to screen empty-string warnings before they reach the descriptor. The session descriptor at `src/interfaces/cli/session.ts:176` uses `if (result.warning !== undefined)` matching the sibling descriptor at `src/interfaces/cli/config.ts:27`. F-003 asked for layer-independent correctness within the handoff flow (descriptor truthy); F-004 asked for sibling-descriptor consistency in the CLI-interface layer (descriptor strict). PR #90 resolves the cycle by keeping the descriptor strict (matches config.ts) and relying on the handler's truthy filter as the defensive screen — strict matches the sibling, truthy at the handler screens empty strings.
- Impact: A future author maintaining the handoff flow sees two layers using different predicates for the same optional field. Without documented intent, the next edit can collapse either side into the other without realising the design split.
- Resolution: Extract the warning emission into the shared `writeWarning` helper proposed in the entry above. A helper that accepts `string | undefined` and short-circuits on `undefined` collapses both the handler filter and the descriptor guard into one site, eliminating both axes of the disagreement. Until that helper exists, the descriptor uses the strict form and the handler keeps the truthy filter as the empty-string screen.
