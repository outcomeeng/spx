# Main Checkout Classifier

Main-checkout designation splits into pure total functions over an observed `GitFacts` value — `isMainCheckout(facts: GitFacts): boolean` and `mainCheckoutPath(facts: GitFacts): string | null` — and an asynchronous probe that gathers those facts through the injected `GitDependencies` git runner. The pure functions own the non-bare-versus-bare decision and the pool's name-and-placement rule of [`spx/15-worktree-management.pdr.md`](../15-worktree-management.pdr.md): a non-bare repository's main checkout is its main working tree — the parent of the common directory — reached from any of its worktrees, and a bare-repository pool's main checkout is the worktree beside the bare repository whose directory name is the `origin` remote's repository name. `isMainCheckout` tests whether the observed checkout is that one, and `mainCheckoutPath` constructs its absolute path — the parent of the common directory for a non-bare repository, or that parent joined with the repository name for a pool, returning null when a pool resolves no repository name. The probe performs the git reads and applies no designation. `GitFacts` carries only git-plumbing observations — the worktree root (`git rev-parse --show-toplevel`), the absolute common directory (`git rev-parse --git-common-dir`), whether the common directory is bare (`git config --get core.bare`), and the `origin` remote URL (`git remote get-url origin`) — so designation reads layout and main-checkout standing from git plumbing alone, independent of any checked-out branch.

## Rationale

The main-checkout rule ranges over input combinations — non-bare versus pool, and within a pool the two signals (repository-name match and sibling placement) each agreeing or disagreeing. A pure classifier over a `GitFacts` record verifies every combination as an `l1` mapping over enumerated values with no git fixture per case, which is the evidence the node's main-checkout mapping assertion requires; a single asynchronous function that read git and decided in one body would force a constructed worktree per combination. Concentrating all derivation — the non-bare-versus-pool distinction and the repository-name extraction included — in the pure functions keeps the probe a thin reader whose only verification burden is reading the right plumbing through the injected runner.

`GitFacts` holds raw observations rather than precomputed values such as a designation boolean or a parsed repository name, because any derivation performed in the probe escapes the pure functions and would demand a git fixture to verify. Bareness and the remote URL are observations, not derivations: a non-bare repository's main checkout is its main working tree whether or not it has linked worktrees, and the structural relationship between the common directory and the worktree cannot tell a non-bare repository's linked worktree from a bare-pool member — only `git config --get core.bare` separates them, reading true from every bare-pool worktree where `--is-bare-repository` reads false. The repository name is parsed from the raw `origin` URL inside the pure functions, so the parse — stripping a trailing `.git` and taking the final path segment across both `https://host/owner/repo.git` and `git@host:owner/repo.git` forms — verifies as enumerated cases without a configured remote per case.

Given bareness and the observed paths, `mainCheckoutPath` reconstructs the designated checkout's location — the parent of the common directory, or for a pool that parent joined with the repository name — rather than enumerating worktrees to find the one that satisfies the rule, so path designation stays pure and fixture-free like the boolean verdict, and a consumer that must name where to run a main-checkout-only operation reads it without re-deriving git topology.

The probe composes the git-detection module's readers — the toplevel, common-dir, bareness, and remote-URL reads — through the `GitDependencies` interface, so main-checkout detection opens no separate git-subprocess boundary and reuses the injection contract [`spx/36-session.enabler/26-worktree-detection.adr.md`](../36-session.enabler/26-worktree-detection.adr.md) and [`spx/16-config.enabler/65-product-directory-api.enabler/21-git-root-result-shape.adr.md`](../16-config.enabler/65-product-directory-api.enabler/21-git-root-result-shape.adr.md) define.

## Invariants

- `isMainCheckout` and `mainCheckoutPath` are total and pure: for every `GitFacts` value each returns its result — a boolean, or an absolute path or null — with no filesystem, process, clock, or network access.
- `isMainCheckout` and `mainCheckoutPath` are deterministic: a given `GitFacts` value yields the same result on every call.
- `mainCheckoutPath` returns null only when no main checkout is designable — a bare-repository pool whose `origin` remote resolves no repository name.
- When `isMainCheckout(facts)` holds, `mainCheckoutPath(facts)` equals `facts.worktreeRoot` — the checkout the rule accepts is the one the path designates.
- The designation depends on no checked-out or default branch: `GitFacts` carries neither, and the verdict is identical across every branch a worktree may hold.
- The probe sets every `GitFacts` field from a git-plumbing read and applies no main-checkout logic.

## Verification

### Audit

- ALWAYS: `isMainCheckout` and `mainCheckoutPath` accept a `GitFacts` value and perform no I/O — git reads happen only in the probe, which gathers `GitFacts` through the injected `GitDependencies` runner ([audit])
- ALWAYS: `mainCheckoutPath` constructs the bare-pool path by inverting the name-and-placement rule — the parent of the common directory joined with the `origin` repository name — and never enumerates worktrees or reads additional git state to locate the main checkout ([audit])
- ALWAYS: the pure functions separate a non-bare repository (main checkout is the main working tree) from a bare-repository pool (main checkout is the repository-named sibling worktree) by the `commonDirIsBare` observation, never by the common-dir-versus-worktree path relationship alone — that relationship cannot tell a non-bare repository's linked worktree from a bare-pool member ([audit])
- ALWAYS: the pure functions derive the pool's main-checkout directory name from the `origin` remote URL carried in `GitFacts`, never from the checked-out branch, the default branch, or a recorded path ([audit])
- ALWAYS: `GitFacts` carries only git-plumbing observations — the worktree root, the absolute common directory, whether the common directory is bare (read from `git config --get core.bare`), and the `origin` remote URL (read from `git remote get-url origin`) — and the probe stores raw reads, deriving no designation and no parsed name ([audit])
- ALWAYS: the probe composes the git-detection module's readers and the `GitDependencies` interface rather than opening a separate git-subprocess boundary ([audit])
- ALWAYS: a consumer obtains the main-checkout verdict or path by composing the probe with `isMainCheckout` or `mainCheckoutPath`, never by re-reading git topology or re-deriving the rule at its call site ([audit])
- NEVER: derive the main checkout or its path from a recorded tool path, a `.git` file-versus-directory test, the checked-out branch, or any signal other than git plumbing — the `origin` repository name, the directory name, the common-dir relationship, and the common-dir bareness, per [`spx/15-worktree-management.pdr.md`](../15-worktree-management.pdr.md) ([audit])
- NEVER: `vi.mock()`, `jest.mock()`, or `memfs` substitute for the git boundary — the probe's tests inject a `GitDependencies` double and the pure functions' tests pass `GitFacts` values directly ([audit])
