# Main Checkout Classifier

Main-checkout designation splits into pure total functions over an observed `GitFacts` value ‚Äî `isMainCheckout(facts: GitFacts): boolean` and `mainCheckoutPath(facts: GitFacts): string | null` ‚Äî and an asynchronous probe that gathers those facts through the injected `GitDependencies` git runner. The pure functions own the non-bare-versus-bare decision and the three-signal pool rule of [`spx/15-worktree-management.pdr.md`](../15-worktree-management.pdr.md): a non-bare repository's main checkout is its main working tree ‚Äî the parent of the common directory ‚Äî reached from any of its worktrees, and a bare-repository pool's main checkout is the default-branch worktree beside the bare repository. `isMainCheckout` tests whether the observed checkout is that one, and `mainCheckoutPath` constructs its absolute path ‚Äî the parent of the common directory for a non-bare repository, or that parent joined with the default branch name for a pool, returning null when a pool resolves no default branch. The probe performs the git reads and applies no designation. `GitFacts` carries only git-plumbing observations ‚Äî the worktree root (`git rev-parse --show-toplevel`), the absolute common directory (`git rev-parse --git-common-dir`), whether the common directory is bare (`git config --get core.bare`), the checked-out branch, and the default branch (`origin/HEAD`'s target) ‚Äî so designation reads layout and main-checkout standing from git plumbing alone.

## Rationale

The main-checkout rule ranges over many input combinations ‚Äî non-bare versus pool, and within a pool the three signals (default-branch match, directory-name match, sibling placement) each agreeing or disagreeing independently. A pure classifier over a `GitFacts` record verifies every combination as an `l1` mapping over enumerated values with no git fixture per case, which is the evidence the node's main-checkout mapping assertion requires; a single asynchronous function that read git and decided in one body would force a constructed worktree per combination. Concentrating all derivation ‚Äî the non-bare-versus-pool distinction included ‚Äî in the classifier keeps the probe a thin reader whose only verification burden is reading the right plumbing through the injected runner.

`GitFacts` holds raw observations rather than precomputed booleans such as an `isMainCheckout` flag, because any derivation performed in the probe escapes the pure functions and would demand a git fixture to verify. Bareness is one such observation, not a derivation: a non-bare repository's main checkout is its main working tree whether or not it has linked worktrees, and the structural relationship between the common directory and the worktree cannot tell a non-bare repository's linked worktree from a bare-pool member ‚Äî only `git config --get core.bare` separates them, reading true from every bare-pool worktree where `--is-bare-repository` reads false. Given bareness, the same observed paths let `mainCheckoutPath` reconstruct the designated checkout's location ‚Äî the parent of the common directory, or for a pool that parent joined with the default branch ‚Äî rather than enumerating worktrees to find the one that satisfies the rule, so path designation stays pure and fixture-free like the boolean verdict, and a consumer that must name where to run a main-checkout-only operation reads it without re-deriving git topology.

The probe composes the git-detection module's readers ‚Äî default-branch resolution, the checked-out-branch reader, and the toplevel and common-dir reads ‚Äî through the `GitDependencies` interface, so main-checkout detection opens no separate git-subprocess boundary and reuses the injection contract [`spx/36-session.enabler/26-worktree-detection.adr.md`](../36-session.enabler/26-worktree-detection.adr.md) and [`spx/16-config.enabler/65-product-directory-api.enabler/21-git-root-result-shape.adr.md`](../16-config.enabler/65-product-directory-api.enabler/21-git-root-result-shape.adr.md) define.

## Invariants

- `isMainCheckout` and `mainCheckoutPath` are total and pure: for every `GitFacts` value each returns its result ‚Äî a boolean, or an absolute path or null ‚Äî with no filesystem, process, clock, or network access.
- `isMainCheckout` and `mainCheckoutPath` are deterministic: a given `GitFacts` value yields the same result on every call.
- `mainCheckoutPath` returns null only when no main checkout is designable ‚Äî a bare-repository pool whose default branch is unresolved.
- When `isMainCheckout(facts)` holds, `mainCheckoutPath(facts)` equals `facts.worktreeRoot` ‚Äî the checkout the rule accepts is the one the path designates.
- The probe sets every `GitFacts` field from a git-plumbing read and applies no main-checkout logic.

## Verification

### Audit

- ALWAYS: `isMainCheckout` and `mainCheckoutPath` accept a `GitFacts` value and perform no I/O ‚Äî git reads happen only in the probe, which gathers `GitFacts` through the injected `GitDependencies` runner ([audit])
- ALWAYS: `mainCheckoutPath` constructs the bare-pool path by inverting the three-signal rule ‚Äî the parent of the common directory joined with the default branch ‚Äî and never enumerates worktrees or reads additional git state to locate the main checkout ([audit])
- ALWAYS: the pure functions separate a non-bare repository (main checkout is the main working tree) from a bare-repository pool (main checkout is the qualifying default-branch worktree) by the `commonDirIsBare` observation, never by the common-dir-versus-worktree path relationship alone ‚Äî that relationship cannot tell a non-bare repository's linked worktree from a bare-pool member ([audit])
- ALWAYS: `GitFacts` carries only git-plumbing observations ‚Äî the worktree root, the absolute common directory, whether the common directory is bare (read from `git config --get core.bare`), the checked-out branch, and the default branch ‚Äî and the probe stores raw reads, deriving no designation ([audit])
- ALWAYS: the probe composes the git-detection module's readers and the `GitDependencies` interface rather than opening a separate git-subprocess boundary ([audit])
- ALWAYS: a consumer obtains the main-checkout verdict or path by composing the probe with `isMainCheckout` or `mainCheckoutPath`, never by re-reading git topology or re-deriving the rule at its call site ([audit])
- NEVER: derive the main checkout or its path from a recorded tool path, a `.git` file-versus-directory test, or any signal other than git plumbing ‚Äî the default branch, the directory name, the common-dir relationship, and the common-dir bareness, per [`spx/15-worktree-management.pdr.md`](../15-worktree-management.pdr.md) ([audit])
- NEVER: `vi.mock()`, `jest.mock()`, or `memfs` substitute for the git boundary ‚Äî the probe's tests inject a `GitDependencies` double and the pure functions' tests pass `GitFacts` values directly ([audit])
