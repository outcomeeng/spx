# Testing Execution Recording and Per-Node Run

Last-run evidence recording and the registry-based per-node run are command-layer handlers under `src/commands/test/` that share one recording path: each resolves the discovered test files for its scope, dispatches the matching files through the testing registry, assembles a `TestRunState`, and persists it through the injected `TestRunStateFileSystem`. The four staleness digests are derived by one shared current-staleness-inputs function that both the recording path and the status resolver's freshness check consume, so a node's recorded staleness inputs and the current inputs later compared against them derive from one recipe. This decision composes only that parent surface; the storage schema, atomic write, digest derivation, and staleness comparison are governed by `spx/41-test.enabler/43-last-run-evidence.enabler/11-last-run-file.adr.md`, `spx/41-test.enabler/43-last-run-evidence.enabler/21-test-state-storage.adr.md`, `spx/41-test.enabler/43-last-run-evidence.enabler/32-terminal-write-protocol.adr.md`, and `spx/41-test.enabler/43-last-run-evidence.enabler/43-staleness-comparison.adr.md`. The four staleness inputs are derived from sources outside this node: the testing config digest from the canonical descriptor digest (`spx/16-config.enabler/54-canonical-descriptor-digest.enabler/canonical-descriptor-digest.md`), the product input digests from root paths declared by the testing language descriptors and covered-test-path-derived inputs those descriptors derive, and the discovered path-set and content digests from the child helpers `digestTestPaths` and `digestTestContents`; the branch name and head SHA are recorded identity fields, not staleness inputs, sourced from the product-directory API (`spx/16-config.enabler/65-product-directory-api.enabler/product-directory-api.md`).

## Rationale

A single recording function consumed by both runs guarantees the two produce identical evidence shape and staleness inputs — a node refreshed individually is indistinguishable from the same node covered by a full run, which is what lets status read either interchangeably. Placing recording in `src/commands/test/` follows the layering in `spx/14-cli-composition.adr.md`: it composes the pure digest helpers and the storage surface with filesystem, clock, and git reads, returns a result, and carries no Commander binding or process exit, so it verifies against temporary fixtures. Dispatching the per-node run through `src/test/registry.ts` keeps the single-node path language-agnostic per `spx/19-language-registration.adr.md`, so the status domain obtains every outcome through one registry-driven path and names no language. Returning the per-node outcome to the caller — rather than having the testing domain reach into status — preserves the composition direction declared by `spx/31-spec-domain.enabler/54-spec-cli-commands.enabler/21-status-testing-delegation.adr.md`: status composes the testing provider through an injected resolver, and the testing domain depends on neither the status nor the spec domain.

Recording inside the pure dispatch (`runTests`) or the testing domain layer was rejected because it drags filesystem, clock, and git I/O into a layer `spx/14-cli-composition.adr.md` keeps free of them; recording inside the Commander descriptor was rejected because it mixes evidence assembly with the process boundary, exercisable only through the built executable; separate recording paths for the full run and the per-node run were rejected because they let the two drift in captured fields or digest derivation, breaking the read-either-interchangeably property; a status-owned per-node runner was rejected because it duplicates registry dispatch and forces status to name languages.

## Invariants

- The full-suite recording and the per-node recording assemble and persist their `TestRunState` through one shared recording function; the scopes differ only by the discovered-file subset they cover.
- A node's recorded staleness inputs and the current staleness inputs the status resolver compares against them are derived by one shared current-staleness-inputs function; the record side and the read side never derive the four digests by separate recipes.
- Every persisted `TestRunState` carries runner outcomes, the four staleness digests (config, path-set, content, product-input), and the branch and head-SHA identity fields.
- The per-node run's result is the node's runner outcome returned to the caller; the testing domain imports neither the status nor the spec domain.

## Verification

### Audit

- ALWAYS: evidence recording and the per-node run live under `src/commands/test/` as handlers that return results and accept filesystem, clock, and git access as injected parameters — so they verify against temporary fixtures without mocking ([audit])
- ALWAYS: the full-suite run and the per-node run assemble and persist their `TestRunState` through one shared recording function — so per-node and full-run evidence are identical in shape and staleness inputs ([audit])
- ALWAYS: the per-node run dispatches through `src/test/registry.ts` and selects the node's tests by its discovered test paths — so the single-node path names no language ([audit])
- ALWAYS: the recording handler derives all four staleness digests (testing config digest, discovered path-set digest, discovered content digest, testing-language product input digests from root and covered-path-derived inputs) and records the branch name and head SHA, persisting through the injected `TestRunStateFileSystem` ([audit])
- ALWAYS: derive the four staleness digests through one shared current-staleness-inputs function that the recording path and the status resolver's freshness check both consume — so freshly recorded evidence is never read as stale through recipe drift ([audit])
- ALWAYS: the per-node run returns the node's runner outcome to its caller for the injected status resolver to consume ([audit])
- NEVER: place evidence recording or per-node run orchestration in `src/domains/test/` or in the Commander descriptor — I/O orchestration belongs to the command layer ([audit])
- NEVER: reference a specific language by name in the per-node run dispatch — every outcome arrives through the registry enumeration ([audit])
- NEVER: access the filesystem, clock, or git through direct `fs`, `Date.now`, or `child_process` calls instead of injected dependencies, or stand in a double via `vi.mock()` / `jest.mock()` ([audit])
- NEVER: the testing domain imports the status or spec domain — the per-node outcome flows outward through the returned result, never through a reverse import ([audit])
