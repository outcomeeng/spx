# Journal Channel

spx presents typed verification runs through the `spx verification` command family and persists every one of them — whichever party drives the run — through the verification run-journal substrate. The `spx journal` verbs — `open`, `append`, `read --from <cursor>`, `seal`, `render`, `list`, and `read-set` — operate one append-only event journal that is a run's sole source of truth, and the journal channel binds both its event store and its streaming surface at the edge from the environment rather than from a verb or flag the caller chooses. By default the event store is a local run-journal file under `.spx/branch/<branch-slug>/<type>/` and each appended event streams to standard output; under continuous integration on a GitHub pull request, the event store persists durably through the GitHub Appendable store — each sealed run retained by the workflow's upload step under its per-run artifact name and the pull request's prior runs restored and hydrated at open — while each append re-renders the current projection to the pull-request comment through the GitHub Snapshot adapter. The journal's `<type>` scope segment is supplied by the `spx verification` command boundary or another substrate caller; the channel implements no verification agent and judges no run.

## Rationale

A verification run is one append-only journal per `spx/15-agent-run-journal.enabler/21-event-sourced-journal.adr.md`: every output surface is a projection rendered from the journal, and whichever party drives the run — an agent, a launcher, or spx executing a runner — emits through one backend-neutral channel that binds the backend at the edge — local file or hosted pull-request comment — swappable without changing the driver. The `spx verification` command family packages that substrate behind the product's public typed lifecycle, while the journal remains the event-store layer underneath it.

The journal channel stays type-agnostic because the event-store operations, backends, scope path shape, and projections are identical for every verification kind. Typed verification vocabulary belongs at the `spx verification` command boundary, where type-specific scope and finding validation happen before an append reaches the journal. Every verification type shares that substrate: a run records scope and finding evidence into the journal whether a caller drove it or spx executed the runner, so the journal carries every type's evidence rather than one verdict mode's, per `spx/34-verification.enabler/verification.md`.

Backend selection lives at the edge, read from the environment inside the channel, because where a run persists and streams is a property of the invocation environment, not of the skill or the verb: a developer checkout wants a local file and standard output, a CI pull-request run wants a durable artifact store and a pull-request comment projection. A `--backend` flag threaded through every verb, or a skill that names its backend, would drift across callers and couple each to one environment. The pull-request comment transport belongs to the Snapshot projection of `spx/21-infrastructure.enabler/43-github-ci.enabler/21-snapshot-adapter.enabler`, while the durable run record is the runner-local file that the Appendable store of `spx/21-infrastructure.enabler/43-github-ci.enabler/21-artifact-journal-store.enabler` defines and names and that the verification workflow's artifact steps retain and restore; the journal command composes those injected capabilities and never owns GitHub API or Actions-artifact mechanics.

Rejected: per-type top-level subcommands (`spx audit`, `spx review`) — they fragment the shared lifecycle and imply spx orchestrates the agent; forcing callers to hand-format raw journal events — it makes every agent or wrapper own protocol details that `spx verification run` can validate; a single end-of-run result instead of a streamed journal — it defeats the incremental observability that is the channel's purpose; and a caller-selected backend — it moves an environment property into skill or verb surface.

## Invariants

- The journal channel carries no verification-type branching in its code; `<type>` is a scope segment supplied by the `spx verification run` boundary or another substrate caller.
- The set of verbs (`open`, `append`, `read --from <cursor>`, `seal`, `render`, `list`, `read-set`) is identical for every verification kind and every backend.
- Backend identity is resolved once, from the environment, inside the channel; no verb argument or skill prose selects it.
- A run's events are its sole source of truth; every rendered surface is a projection of an event prefix.
- Type-specific scope and finding validation happens before a caller appends through the journal substrate.

## Verification

### Audit

- ALWAYS: the journal channel resolves its event store and streaming surface from the environment — the local Appendable store with standard output by default, the GitHub Appendable store with the Snapshot pull-request comment under continuous integration — never from a verb argument, a flag, or skill prose ([audit])
- ALWAYS: the verbs `open`, `append`, `read --from <cursor>`, `seal`, `render`, `list`, and `read-set` operate one append-only event journal bound to the agent-run-journal contract of `spx/15-agent-run-journal.enabler`, and every rendered surface is a projection of the event history ([audit])
- ALWAYS: `spx verification run` owns public verification-run type, scope, and finding validation before appending to the journal substrate ([audit])
- ALWAYS: a run's local persistence resolves under `.spx/branch/<branch-slug>/<type>/` at the Git common-dir product root per `spx/15-worktree-management.pdr.md`, with `<branch-slug>` from the state-store branch slug and `<type>` an opaque caller-supplied segment ([audit])
- ALWAYS: under the GitHub pull-request environment, a run's events persist durably through the GitHub Appendable store of `spx/21-infrastructure.enabler/43-github-ci.enabler/21-artifact-journal-store.enabler` and each append re-renders the event-history projection through `spx/21-infrastructure.enabler/43-github-ci.enabler/21-snapshot-adapter.enabler` to the pull-request comment, so the run persists across the ephemeral runner and stays observable on the pull request as it advances ([audit])
- NEVER: journal domain or command code branches on a verification-type name (`audit`, `review`) — type-specific behavior belongs at the `spx verification run` boundary ([audit])
- NEVER: the journal channel implements a verification agent or judges a run — whichever party drives a run calls the channel, and the channel records and streams ([audit])
- NEVER: a verb argument, flag, or skill prose selects the backend — backend binding is an edge concern read from the environment ([audit])
