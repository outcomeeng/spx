# Main Checkout Classifier

Main-checkout designation splits into pure total functions over an observed `GitFacts` value â€” `isMainCheckout(facts: GitFacts): boolean` and `mainCheckoutPath(facts: GitFacts): string | null` â€” and an asynchronous probe that gathers those facts through the injected `GitDependencies` git runner. The pure functions own the non-bare-versus-bare decision and the pool's name-and-placement rule of [`spx/15-worktree-management.pdr.md`](../../15-worktree-management.pdr.md): a non-bare repository's main checkout is its main working tree â€” the parent of the common directory â€” reached from any of its worktrees, and a bare-repository pool's main checkout is the worktree beside the bare repository whose directory name is the `origin` remote's repository name. `isMainCheckout` tests whether the observed checkout is that one, and `mainCheckoutPath` designates its absolute path â€” the parent of the common directory for a non-bare repository, or that parent joined with the repository name for a pool only when that worktree appears in the observed worktree list, returning null when a pool resolves no repository name or lacks the repository-named worktree. The probe performs the git reads and applies no designation. `GitFacts` carries only git-plumbing observations â€” the worktree root (`git rev-parse --show-toplevel`), the observed worktree roots and successful-read marker (`git worktree list --porcelain`), the absolute common directory (`git rev-parse --git-common-dir`), whether the common directory is bare (`git config --get core.bare`), and the `origin` remote URL (`git remote get-url origin`) â€” so designation reads layout and main-checkout standing from git plumbing alone, independent of any checked-out branch.

## Rationale

The main-checkout rule ranges over input combinations â€” non-bare versus pool, and within a pool the two signals (repository-name match and sibling placement) each agreeing or disagreeing. A pure classifier over a `GitFacts` record verifies every combination as an `l1` mapping over enumerated values with no git fixture per case, which is the evidence the node's main-checkout mapping assertion requires; a single asynchronous function that read git and decided in one body would force a constructed worktree per combination. Concentrating all derivation â€” the non-bare-versus-pool distinction and the repository-name extraction included â€” in the pure functions keeps the probe a thin reader whose only verification burden is reading the right plumbing through the injected runner.

`GitFacts` holds raw observations rather than precomputed values such as a designation boolean or a parsed repository name, because any derivation performed in the probe escapes the pure functions and would demand a git fixture to verify. Bareness, the worktree list, the worktree-list read result, and the remote URL are observations, not derivations: a non-bare repository's main checkout is its main working tree whether or not it has linked worktrees, the structural relationship between the common directory and the worktree cannot tell a non-bare repository's linked worktree from a bare-pool member â€” only `git config --get core.bare` separates them, reading true from every bare-pool worktree where `--is-bare-repository` reads false â€” and a bare-pool main-checkout path exists only when git reports the repository-named worktree. A bare-pool checkout whose own root is absent from the observed worktree list is internally inconsistent and fails the main-checkout verdict, keeping `isMainCheckout` and `mainCheckoutPath` aligned for every type-valid `GitFacts`. A bare-pool checkout whose worktree-list read fails lacks the positive observed-root signal and fails the pure main-checkout verdict; hook consumers that need fail-open behavior handle that policy at their own boundary. The repository name is parsed from the raw `origin` URL inside the pure functions, so the parse â€” stripping a trailing `.git` and taking the final path segment across both `https://host/owner/repo.git` and `git@host:owner/repo.git` forms â€” verifies as enumerated cases without a configured remote per case.

Given bareness, the observed paths, and the observed worktree list, `mainCheckoutPath` designates the checkout's location â€” the parent of the common directory, or for a pool that parent joined with the repository name only when that candidate is in the observed worktree list â€” so path designation stays pure like the boolean verdict, and a consumer that must name where to run a main-checkout-only operation reads it without re-deriving git topology.

The probe composes the git-detection module's readers â€” the toplevel, common-dir, bareness, and remote-URL reads â€” through the `GitDependencies` interface, so main-checkout detection opens no separate git-subprocess boundary and reuses the injected-dependency contract and product-root result shape [`spx/17-state.adr.md`](../../17-state.adr.md) defines.

## Invariants

- `isMainCheckout` and `mainCheckoutPath` are total and pure: for every `GitFacts` value each returns its result â€” a boolean, or an absolute path or null â€” with no filesystem, process, clock, or network access.
- `isMainCheckout` and `mainCheckoutPath` are deterministic: a given `GitFacts` value yields the same result on every call.
- `mainCheckoutPath` returns null only when no main checkout is designable â€” a bare-repository pool whose `origin` remote resolves no repository name or whose repository-named worktree is absent.
- When `isMainCheckout(facts)` holds, `mainCheckoutPath(facts)` equals `facts.worktreeRoot` â€” the checkout the rule accepts is the one the path designates.
- The designation depends on no checked-out or default branch: `GitFacts` carries neither, and the verdict is identical across every branch a worktree may hold.
- The probe sets every `GitFacts` field from a git-plumbing read and applies no main-checkout logic.

## Verification

### Audit

- ALWAYS: `isMainCheckout` and `mainCheckoutPath` accept a `GitFacts` value and perform no I/O â€” git reads happen only in the probe, which gathers `GitFacts` through the injected `GitDependencies` runner ([audit])
- ALWAYS: `mainCheckoutPath` constructs the bare-pool candidate by inverting the name-and-placement rule â€” the parent of the common directory joined with the `origin` repository name â€” and returns it only when that candidate appears in `GitFacts.worktreeRoots` ([audit])
- ALWAYS: the pure functions separate a non-bare repository (main checkout is the main working tree) from a bare-repository pool (main checkout is the repository-named sibling worktree) by the `commonDirIsBare` observation, never by the common-dir-versus-worktree path relationship alone â€” that relationship cannot tell a non-bare repository's linked worktree from a bare-pool member ([audit])
- ALWAYS: the pure functions derive the pool's main-checkout directory name from the `origin` remote URL carried in `GitFacts`, never from the checked-out branch, the default branch, or a recorded path ([audit])
- ALWAYS: `GitFacts` carries only git-plumbing observations â€” the worktree root, observed worktree roots and successful-read marker, the absolute common directory, whether the common directory is bare (read from `git config --get core.bare`), and the `origin` remote URL (read from `git remote get-url origin`) â€” and the probe stores raw reads, deriving no designation and no parsed name ([audit])
- ALWAYS: the probe composes the git-detection module's readers and the `GitDependencies` interface rather than opening a separate git-subprocess boundary ([audit])
- ALWAYS: a consumer obtains the main-checkout verdict or path by composing the probe with `isMainCheckout` or `mainCheckoutPath`, never by re-reading git topology or re-deriving the rule at its call site ([audit])
- NEVER: derive the main checkout or its path from a recorded tool path, a `.git` file-versus-directory test, the checked-out branch, or any signal other than git plumbing â€” the `origin` repository name, the directory name, the common-dir relationship, and the common-dir bareness, per [`spx/15-worktree-management.pdr.md`](../../15-worktree-management.pdr.md) ([audit])
- NEVER: `vi.mock()`, `jest.mock()`, or `memfs` substitute for the git boundary â€” the probe's tests inject a `GitDependencies` double and the pure functions' tests pass `GitFacts` values directly ([audit])
